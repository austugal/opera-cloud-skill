# Interfaces, IFC8, and OXI

## Interface types in OPERA

OPERA connects to many on-property and external systems through interfaces. Common categories:

- Payment interfaces (card authorisation and settlement through a payment platform).
- Point of sale (POS) for posting food, beverage, and retail charges to guest folios.
- Door lock / key systems.
- Telephone / call accounting.
- Voice mail, minibar, pay TV / in-room entertainment, and similar in-room services.
- Back-office finance exports (handled as exports rather than live interfaces; see
  `backoffice-export.md`).
- Channel and distribution connectivity (handled through Distribution APIs and OXI; see
  `distribution-channels.md`).

## IFC8

IFC8 is Oracle Hospitality's interface platform/controller used to connect OPERA to external
on-property systems through standard interface protocols. Concepts:

- IFC8 mediates between OPERA and a device or system, translating messages both ways.
- An interface instance is configured with the connection parameters and the message behaviour for
  the specific external system.
- Authentication parameters configured on the OPERA side must match those configured on the
  interface side; a mismatch is a common cause of an interface failing to start or link.
- Interfaces are licensed and enabled per property; an interface must be both configured and active
  to pass traffic.

Operational notes (observed and documented behaviour):
- When an interface will not link, first verify the connection parameters and the matching
  authentication values on both sides before suspecting the network.
- Restarting an interface controller clears a hung link in many cases, but recurring drops point to
  a configuration or network problem that a restart only masks.

## OXI / OPERA Exchange Interface

OXI (OPERA Exchange Interface) is the translation and exchange layer OPERA uses to communicate with
external reservation and distribution systems. It maps external code sets to OPERA code sets and
governs how messages are exchanged.

Concepts:
- Translation tables map external values (rate codes, room types, market segments, source codes) to
  OPERA values and back.
- During a channel manager migration, the OXI translation layer is what gets reconfigured while the
  external codes typically stay the same (see `distribution-channels.md`).
- OXI message flow covers both inbound (reservations from the external system) and outbound
  (availability, rates, restrictions to the external system).

## Interface configuration in OPERA Cloud (INT / INTCFG)

OPERA Cloud exposes interface configuration through the INT / INTCFG module surface and the
Administration interface menus. Use these to define and query interface definitions. Confirm the
exact configuration path and available fields against the documentation for the running release.

## Practical interface troubleshooting checklist

1. Is the interface licensed and enabled for the property?
2. Do the authentication / connection parameters match on both the OPERA side and the external side?
3. Is the interface controller running and the link reported up?
4. Are messages being generated by the triggering action (for example a POS charge) at all?
5. If messages generate but do not arrive, is it a translation/mapping gap or a transport problem?
6. Check the interface logs on both sides for the specific rejected message before changing
   configuration.
